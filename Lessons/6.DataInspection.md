## 1. What is a Tibble?

A **tibble** is the tidyverse's modern version of a data frame.  
It stores data in the same rows-and-columns structure, but behaves more safely and prints more clearly.

### data.frame vs tibble ‚Äî the key differences

| | `data.frame` | `tibble` |
|---|---|---|
| **Prints** | Dumps every row to the console | Shows first 10 rows, fits to screen width |
| **Column types shown** | No | Yes ‚Äî shown under each column name |
| **Strings to factors** | Depends on R version / settings | Never ‚Äî always stays as character |
| **Partial column name matching** | Yes ‚Äî `df$sal` finds `salary` silently | No ‚Äî must use the exact name |
| **Row names** | Supported | Not used |
| **`[` subsetting** | Can silently return a vector | Always returns a tibble |
| **Created by** | `data.frame()` / `read.csv()` | `tibble()` / `read_csv()` |

### Why does partial matching matter?

```r
# data.frame ‚Äî this works silently, even if you made a typo
df_base <- data.frame(salary = c(72000, 65000), status = c("FT", "PT"))
df_base$sal       # Returns the salary column ‚Äî no warning!

# tibble ‚Äî this fails loudly
df_tbl <- tibble(salary = c(72000, 65000), status = c("FT", "PT"))
df_tbl$sal        # Returns NULL with a warning ‚Äî protects you from silent bugs
```

> **Bottom line:** tibbles are stricter than data frames.  
> They refuse to guess what you meant, which catches errors early.

## 4. Checking Data Types

### Why types matter so much

- R performs **different operations** depending on type ‚Äî adding two numbers works; adding two strings fails
- A column stored as the **wrong type** will silently produce wrong results or errors later
- **Numeric stored as character** is one of the most common and dangerous data problems

### The type system ‚Äî what you will encounter

| Type | Shows as | Stores | Example |
|---|---|---|---|
| `double` | `<dbl>` | Decimal numbers | `72000`, `3.14` |
| `integer` | `<int>` | Whole numbers | `1L`, `42L` |
| `character` | `<chr>` | Text strings | `"Alice"`, `"Finance"` |
| `logical` | `<lgl>` | TRUE / FALSE | `TRUE`, `FALSE` |
| `Date` | `<date>` | Calendar dates | `2024-03-15` |
| `POSIXct` | `<dttm>` | Date + time | `2024-03-15 09:30:00` |
| `factor` | `<fct>` | Categories with levels | `"High"`, `"Medium"`, `"Low"` |

### Checking types

```r
# Option 1 ‚Äî glimpse() (recommended ‚Äî shows all at once)
glimpse(employees)

# Option 2 ‚Äî str() (base R equivalent)
str(employees)

# Option 3 ‚Äî check one column at a time
class(employees$salary)       # [1] "numeric"
class(employees$hire_date)    # [1] "character"  ‚Üê problem!
class(employees$performance)  # [1] "character"  ‚Üê should probably be a factor

# Option 4 ‚Äî check all columns at once
sapply(employees, class)
# employee_id  first_name   last_name  department      salary   hire_date
#   "numeric" "character" "character" "character"   "numeric" "character"
#         age performance
#   "numeric" "character"
```

### Spotting type problems in a tibble

The tibble printout is your first line of defence:

```
$ hire_date   <chr> "2019-03-15", ...    ‚Üê <chr> when it should be <date>
$ performance <chr> "High", "Medium", ...‚Üê <chr> when it should be <fct>
$ employee_id <dbl> 1, 2, 3, ...        ‚Üê <dbl> when <int> would be more precise
```

## 5. The Type Coercion Problem

### What is type coercion?

**Coercion** is when R (or your CSV file) forces a value into a type it was not meant to be.  
This happens silently and is one of the most common sources of bugs in data analysis.

### The most dangerous case ‚Äî numeric read as character

This happens when:
- A numeric column contains **even one non-numeric value** (e.g. `"N/A"`, `"PENDING"`, `""`)
- `read_csv()` encounters that value and **falls back to reading the whole column as character**
- Your salary or age column is now text ‚Äî and you won't know until something breaks

```r
# Simulate reading a CSV where salary contains a text value
employees_problem <- read_csv(
  "employee_id,first_name,salary
   1,Alice,72000
   2,Bob,N/A
   3,Carol,95000",
  show_col_types = FALSE
)

glimpse(employees_problem)
# $ employee_id <dbl> 1, 2, 3
# $ first_name  <chr> "Alice", "Bob", "Carol"
# $ salary      <chr> "72000", "N/A", "95000"   ‚Üê entire column is now character!
```

### Why is this dangerous?

```r
# This silently gives the wrong answer ‚Äî or an error
mean(employees_problem$salary)
# Warning: argument is not numeric or logical: returning NA

# Comparisons fail silently
employees_problem$salary > 70000
# Error: comparison is not allowed for character vectors

# Sorting gives alphabetical order, not numeric
sort(employees_problem$salary)
# [1] "72000" "95000" "N/A"   ‚Üê sorted as text, not numbers!
```

### How to detect it

```r
# Always run glimpse() or sapply(df, class) after loading
glimpse(employees_problem)   # spot <chr> where you expect <dbl>

# Or check a specific column
is.numeric(employees_problem$salary)    # FALSE ‚Äî problem confirmed
is.character(employees_problem$salary)  # TRUE
```

### How to fix it

```r
# Step 1 ‚Äî find out what non-numeric values are lurking
unique(employees_problem$salary)
# [1] "72000" "N/A"   "95000"

# Step 2 ‚Äî re-read with those values declared as NA
employees_fixed <- read_csv(
  "employee_id,first_name,salary
   1,Alice,72000
   2,Bob,N/A
   3,Carol,95000",
  na = c("", "NA", "N/A"),
  show_col_types = FALSE
)

glimpse(employees_fixed)
# $ salary <dbl> 72000, NA, 95000   ‚Üê now numeric, with NA for the missing value
```

### Fixing types after the fact ‚Äî `as.*()` conversion functions

Sometimes you have already loaded the data and need to convert types in place:

```r
# Convert character to numeric
employees$salary <- as.numeric(employees$salary)

# Convert character to integer
employees$employee_id <- as.integer(employees$employee_id)

# Convert character to Date
employees$hire_date <- as.Date(employees$hire_date, format = "%Y-%m-%d")

# Convert character to factor (categorical)
employees$performance <- as.factor(employees$performance)
employees$department  <- as.factor(employees$department)

# Verify the fixes
glimpse(employees)
# $ hire_date   <date> 2019-03-15, 2021-07-01, ...  ‚Üê now a proper Date
# $ performance <fct>  High, Medium, High, ...       ‚Üê now a factor
```

### The `as.*()` function family ‚Äî quick reference

| Function | Converts to | Watch out for |
|---|---|---|
| `as.numeric()` | double / numeric | Non-numeric strings become `NA` with a warning |
| `as.integer()` | integer | Decimals are truncated, not rounded |
| `as.character()` | character | Always works ‚Äî everything can become text |
| `as.logical()` | TRUE / FALSE | Only works on `0`/`1` or `"TRUE"`/`"FALSE"` strings |
| `as.Date()` | Date | Must supply `format =` if not `YYYY-MM-DD` |
| `as.factor()` | factor | Levels are set alphabetically by default |

> **Key rule:** If `as.numeric()` produces `NA` values that were not `NA` before,  
> it means there were non-numeric strings in the column. Find them with `unique()` first.

```r
# Safe pattern ‚Äî check before converting
x <- c("72000", "65000", "N/A", "95000")

# Find the problem values
x[is.na(as.numeric(x))]
# [1] "N/A"   ‚Üê now you know what to clean

# Fix them
x[x == "N/A"] <- NA
as.numeric(x)
# [1] 72000 65000    NA 95000
```

## 6. Checking for Missing Values

### Why missing values matter

- Many R functions return `NA` if the input contains `NA` ‚Äî unless you tell them to ignore it
- Missing values can **bias your analysis** if they are not random
- You need to know **how many**, **which columns**, and **whether there is a pattern**

### Counting missing values

```r
# Is a specific value missing?
is.na(employees$department)
# [1] FALSE FALSE FALSE FALSE FALSE  TRUE FALSE FALSE

# Total NAs in one column
sum(is.na(employees$department))
# [1] 1

# Total NAs across every column
colSums(is.na(employees))
# employee_id  first_name   last_name  department      salary
#           0           0           0           1           0
#   hire_date         age performance
#           0           0           2

# Total NAs in the entire dataset
sum(is.na(employees))
# [1] 3

# Percentage missing per column
round(colMeans(is.na(employees)) * 100, 1)
# department  performance
#       12.5         25.0    ‚Üê 25% of performance values are missing
```

### Finding the rows that contain NAs

```r
# Which rows have at least one NA anywhere?
employees[!complete.cases(employees), ]
# Returns rows 6, 7, and 8

# complete.cases() returns TRUE for rows with no NAs at all
complete.cases(employees)
# [1]  TRUE  TRUE  TRUE  TRUE  TRUE FALSE FALSE FALSE

# How many complete rows do you have?
sum(complete.cases(employees))
# [1] 5
```

### Visualising missing values (optional ‚Äî requires naniar)

```r
# install.packages("naniar")
library(naniar)

# Quick visual summary of missingness
vis_miss(employees)

# Which combinations of columns tend to be missing together?
gg_miss_upset(employees)
```

### What to do about missing values

This is a decision that depends on your data and your analysis ‚Äî there is no single right answer:

| Option | When to use it |
|---|---|
| **Leave as NA** | Functions like `mean(x, na.rm = TRUE)` handle it; missingness itself is informative |
| **Drop the rows** | `drop_na()` or `na.omit()` ‚Äî only if missingness is rare and random |
| **Impute (fill in)** | Replace with mean, median, or a model prediction ‚Äî for more advanced analysis |
| **Flag the missingness** | Add a new column `is_missing_performance` ‚Äî preserves the information |

```r
library(dplyr)
library(tidyr)

# Option 1 ‚Äî drop rows with any NA
employees_complete <- employees |> drop_na()
nrow(employees_complete)   # [1] 5

# Option 2 ‚Äî drop rows with NA in a specific column only
employees_complete <- employees |> drop_na(performance)

# Option 3 ‚Äî flag missing performance instead of dropping
employees <- employees |>
  mutate(performance_missing = is.na(performance))
```

## 7. Checking for Duplicates

### Why duplicates are dangerous

- A duplicate row gets **counted twice** in every summary and aggregation
- They are easy to miss ‚Äî the data looks fine, the numbers are just slightly off
- Often introduced during **data merges** or when files are exported multiple times

### Detecting duplicates

```r
# Are there any duplicate rows at all?
anyDuplicated(employees)
# [1] 8   ‚Üê row 8 is a duplicate of a previous row (non-zero means yes)

# Which rows are duplicates?
duplicated(employees)
# [1] FALSE FALSE FALSE FALSE FALSE FALSE FALSE  TRUE
#                                                 ‚Üë row 8 is the duplicate

# See the actual duplicate rows
employees[duplicated(employees), ]
# Returns row 8 ‚Äî Grace Lee's second entry

# See both the original and its duplicate together
employees[duplicated(employees) | duplicated(employees, fromLast = TRUE), ]
# Returns rows 7 and 8
```

### Checking for duplicates on a key column only

Sometimes a full row is not duplicated, but a key identifier (like `employee_id`) should be unique:

```r
# Are all employee IDs unique?
any(duplicated(employees$employee_id))
# [1] TRUE   ‚Üê there are duplicate IDs

# Which IDs are duplicated?
employees$employee_id[duplicated(employees$employee_id)]
# [1] 7

# How many times does each ID appear?
table(employees$employee_id)
#  1  2  3  4  5  6  7
#  1  1  1  1  1  1  2   ‚Üê ID 7 appears twice
```

### Removing duplicates

```r
# Remove fully duplicated rows (keeps the first occurrence)
employees_clean <- employees |> distinct()
nrow(employees_clean)   # [1] 7

# Remove duplicates based on a key column only
employees_clean <- employees |> distinct(employee_id, .keep_all = TRUE)
```

## 8. Inspection Cheat Sheet

Run these in order every time you load a new dataset:

```r
library(readr)
library(dplyr)

df <- read_csv("your_file.csv")

# ‚îÄ‚îÄ Step 1: First look ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
glimpse(df)              # rows, columns, types, sample values
head(df)                 # first 6 rows
tail(df)                 # last 6 rows
dim(df)                  # dimensions: rows √ó columns

# ‚îÄ‚îÄ Step 2: Check types ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
sapply(df, class)        # type of every column
# Flag: any <chr> that should be <dbl>, <date>, or <fct>?

# ‚îÄ‚îÄ Step 3: Fix wrong types ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
df$date_col   <- as.Date(df$date_col, format = "%Y-%m-%d")
df$num_col    <- as.numeric(df$num_col)
df$cat_col    <- as.factor(df$cat_col)

# ‚îÄ‚îÄ Step 4: Check for missing values ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
colSums(is.na(df))                         # NAs per column
round(colMeans(is.na(df)) * 100, 1)        # % missing per column
sum(complete.cases(df))                    # fully complete rows

# ‚îÄ‚îÄ Step 5: Check for duplicates ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
anyDuplicated(df)                          # 0 = no duplicates
df[duplicated(df), ]                       # see the duplicate rows
any(duplicated(df$id_column))             # check key column

# ‚îÄ‚îÄ Step 6: Quick summary ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
summary(df)              # min, max, mean, NA count for each column
```
